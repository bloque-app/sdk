# Add `us2-account` Support To The SDK

## Goal

Expose the new `us2-account` medium in the SDK without breaking existing Bridge-backed `us-account` support. The SDK must support both media side by side during the soft transition.

This first implementation pass is intentionally limited to `us2-account` accounts of type `individual`. `business` support is deferred.

## What We Now Understand

### Backend reality

- `us2-account` is a distinct medium, not a rename of `us-account`.
- Its create endpoint is `POST /api/mediums/us2-account`.
- It does **not** have a ToS-link or `signed_agreement_id` flow.
- It uses KYC/compliance-driven onboarding and asynchronous virtual-account provisioning.
- Returned details differ from `us-account`.

### `us2-account` input shape

Top-level create request matches other mediums:

- `holder_urn`
- `input`
- optional `webhook_url`
- optional `ledger_account_id`
- optional `metadata`

`input` for `us2-account` currently supports:

- `type: 'individual' | 'business'`
- `email`
- optional `phone`
- optional `tax_id`
- optional `address` with:
  - `street`
  - `city`
  - `state`
  - `postal_code`
  - `country`
- optional `proof_of_address`
- optional `business_formation_document`

Backend-enforced onboarding constraints:

- individual accounts require completed compliance docs and `proof_of_address`
- business accounts require `proof_of_address` and `business_formation_document`

First-pass SDK scope:

- support only `type: 'individual'`
- require `proofOfAddress` in the public SDK input
- do not expose `business`-only fields yet

### `us2-account` details shape

The medium-specific `details` returned by the backend expose:

- `id`
- `user_id`
- optional `virtual_account_id`
- `type` (currently `US_BANK`)
- `currency`
- optional `source_deposit_instructions` with possible fields:
  - `currency`
  - `bank_name`
  - `bank_address`
  - `bank_routing_number`
  - `bank_account_number`
  - `bank_beneficiary_name`
  - `bank_beneficiary_address`
  - `clabe`

Important backend behavior:

- `evm_address` exists internally but is intentionally stripped from API responses.

### `us2-account` lifecycle

- create returns an account before the virtual account is fully ready
- status may progress through:
  - `creation_in_progress`
  - `active`
  - `creation_failed`
  - `disabled`
- `virtual_account_id` and `source_deposit_instructions` may appear only after async activation or later sync/read

### Existing SDK constraints and mismatches

- The SDK already has a dedicated `accounts.us` client for Bridge `us-account`.
- `AccountsClient._mapByMedium()` currently knows only `us-account`, not `us2-account`.
- The shared account medium unions do not include `us2-account`.
- CLI account tooling hardcodes medium enums and currently omits some existing media already.
- `AccountsClient.list()` currently ignores `params.medium` and `params.urn`, which is a pre-existing bug and matters for exposing a new medium cleanly.
- Several mapped account types do not include a `medium` property even though CLI code assumes it exists.

## Recommended SDK Shape

Use a separate client surface:

- keep existing Bridge support as `accounts.us`
- add Kira support as `accounts.us2`

This is the cleanest way to preserve both media during the transition and matches the backend naming strategy.

## Implementation Plan

### 1. Add raw wire support for `us2-account`

Files:

- `packages/accounts/src/internal/wire-types.ts`

Changes:

- extend the raw account `medium` union with `'us2-account'`
- add raw request and response types for Kira-backed account details
- model `source_deposit_instructions` explicitly instead of leaving it as `unknown`

### 2. Add public `us2` SDK types

Files:

- `packages/accounts/src/us2/types.ts` new
- `packages/accounts/src/types.ts`

Changes:

- define `CreateUs2AccountParams` for `individual` only
- define `Us2Account`
- define `ListUs2AccountsParams` and `ListUs2AccountsResult`
- extend `AccountMedium` with `'us2-account'`
- extend `ListAccountsResult` unions with `Us2Account`

Design notes:

- do not reuse the Bridge `UsAccount` type because the fields and onboarding contract differ materially
- expose camelCase fields in the public type, mirroring the rest of the SDK
- for now, narrow the create params to the subset we trust for `individual`

Recommended first-pass `CreateUs2AccountParams`:

- `holderUrn?`
- `type: 'individual'`
- `email`
- `phone?`
- `taxId?`
- `address?` with:
  - `street`
  - `city`
  - `state`
  - `postalCode`
  - `country`
- `proofOfAddress`
- `name?`
- `webhookUrl?`
- `ledgerId?`
- `metadata?`

### 3. Add a dedicated `Us2Client`

Files:

- `packages/accounts/src/us2/us2-client.ts` new
- `packages/accounts/src/accounts-client.ts`
- `packages/accounts/src/index.ts`

Changes:

- add `accounts.us2`
- implement a mapper from wire `us2-account` details to public `Us2Account`
- add `AccountsClient` property construction and `_mapByMedium()` dispatch for `'us2-account'`
- export the new client and types from the package entrypoint

Recommended method surface for `Us2Client`:

- `create(params, options?)`
- `list(params?)`
- `updateMetadata(params)`
- `activate(urn)`
- `freeze(urn)`
- `disable(urn)`

Do not implement `getTosLink()` for `us2`, because the backend has no such endpoint.

### 4. Fix shared account-list behavior while touching the surface

Files:

- `packages/accounts/src/accounts-client.ts`

Changes:

- make `AccountsClient.list()` actually forward supported filters:
  - `holderUrn`
  - `urn`
  - `medium`
- keep the change minimal and aligned with the existing backend query contract

Rationale:

- this is already broken today
- exposing `us2-account` through the generic list surface without fixing filters will make CLI and SDK behavior inconsistent

### 5. Normalize mapped accounts enough for CLI compatibility

Files:

- existing medium mappers under `packages/accounts/src/**`

Changes:

- ensure mapped account objects expose `medium` consistently, including `us2`

Rationale:

- CLI workflows already branch on `account.medium`
- without this, generic listing and overview behavior remain brittle

### 6. Update CLI/MCP surfaces if we want parity there

Files:

- `packages/cli/src/mcp/tools/primitives/accounts.ts`
- `packages/cli/src/mcp/tools/workflows/overview.ts`
- `packages/cli/src/mcp/tools/primitives/us.ts` or a new `us2.ts`
- `packages/cli/src/mcp/server.ts`

Changes:

- extend generic medium enums with `'us2-account'`
- add grouping and rendering for `us2-account`
- optionally add dedicated tools:
  - `create_us2_account`
  - `list_us2_accounts`

Recommendation:

- add generic account listing support for `us2-account` in this same change if CLI is included
- if we add dedicated MCP primitives now, expose only `individual` creation for `us2-account`

### 7. Update docs and examples

Files likely affected:

- `website/en/guide/accounts/*.mdx`
- `website/es/guide/accounts/*.mdx`
- `website/en/guide/features/api-reference.mdx`
- `website/es/guide/features/api-reference.mdx`
- `packages/cli/README.md`

Changes:

- document `accounts.us2`
- describe its async onboarding and bank instruction availability timing
- explicitly state that `us-account` and `us2-account` both exist during the transition
- explicitly state that the initial SDK support is `individual` only and `business` is not yet supported

Recommendation:

- keep docs separate from `accounts.us` because the two flows differ too much

## Verification Plan

After implementation:

- run `bun run --filter @bloque/sdk-accounts typecheck`
- run `bun run --filter @bloque/sdk typecheck`
- run `bun run --filter @bloque/cli build` if CLI is touched
- run `bun run build`

Targeted behavioral checks:

- `accounts.us` still points to `/api/mediums/us-account`
- `accounts.us2` points to `/api/mediums/us2-account`
- generic `accounts.get()` can map both `us-account` and `us2-account`
- generic `accounts.list({ medium: 'us2-account' })` filters correctly
- `Us2Account` mapping preserves bank instruction fields when present
- no `getTosLink()` exists on `accounts.us2`
- `accounts.us2.create()` only accepts `type: 'individual'`

## Risks

- The backend allows generic status updates, but Kira itself does not appear to support meaningful provider-backed freeze/reactivate operations. The SDK can mirror the existing generic status pattern, but those methods should be documented carefully.
- `us2-account` is async by design, so examples and docs must not imply that routing/account details always exist immediately after `create()`.
- If CLI parity is included, medium handling should be updated consistently to avoid another partial-medium rollout.
- The backend technically accepts `business`, but the SDK will intentionally reject or omit it in this pass to avoid shipping an unproven contract.

## Finalized Direction

- expose this as `accounts.us2` and keep `accounts.us` untouched
- support only `individual` creation in the first pass
- defer `business` until the backend contract and operational flow are better proven

## Remaining Product Choice

Implementation can proceed in either of these scopes:

- minimal: `packages/accounts` plus exported SDK surface
- broader: minimal scope plus CLI/MCP parity and docs

Recommended execution order:

- ship the minimal SDK surface first
- add CLI/MCP and docs immediately after if needed
